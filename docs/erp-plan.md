# Mania Group — Максимальний план ERP

Повна дорожня карта ERP-системи магазину брендового одягу. Прив'язана до реального
стану коду (PostgreSQL, Next.js 16, `/erp`). Кожен модуль позначений статусом:
**✅ зроблено · 🔜 наступне · ◻️ заплановано**.

> Принцип: ERP — єдине джерело правди по **залишках, собівартості та грошах**.
> Вітрина (`/`) і адмінка (`/admin`) — споживачі цих даних.

---

## Поточний стан (фундамент готовий)

| Модуль | Що є | Файли |
|---|---|---|
| **Огляд** ✅ | Дашборд: вартість залишку (собівартість+роздріб), маржа, покриття собівартості, звірка дрейфу, low-stock, рух, топ-постачальники | `ErpOverview.tsx`, `lib/erpDashboard.ts` |
| **Товари** ✅ | Матриця розмірів (`product_variants`), леджер руху (`stock_movements`), ручне коригування, +/− | `ErpWorkspace.tsx`, `lib/erp.ts` |
| **Прихід (E4)** ✅ | Прихідні документи, weighted-average собівартість → `products.cost_price`, проведення | `ErpReceiving.tsx`, `lib/receiving.ts` |
| **Постачальники (E4)** ✅ | Довідник + закупівельна аналітика, прив'язка до приходів | `ErpSuppliers.tsx`, `lib/suppliers.ts` |
| **Канали / Вигрузки (E5)** ✅ | Експорт прайс+залишок: CSV, XLSX, Prom.ua, Rozetka YML, Google Merchant | `ErpChannels.tsx`, `lib/channelExport.ts` |
| **Фінанси (рушій)** ✅ | `costSql` (собівартість шарами), `cost_rules` по брендах, `expenses` | `lib/finance.ts` |

Таблиці: `products, product_variants, stock_movements, receipts, receipt_items, suppliers, cost_rules, expenses, orders, order_items, order_events`.

**⚠️ Головна діра зараз:** поштучні залишки (`stock_qty`) = 0 у всіх XLS-товарів
(вигрузки дають лише «в наявності» + розміри, не кількість). Через це вартість
залишку та обороти = 0. Фаза A це закриває.

---

## Фаза A — Реальні залишки (фундамент, розблоковує всі цифри) 🔜

Без реальних кількостей дашборд, обороти й точні вигрузки порожні. Найвищий пріоритет.

- **A1. Імпорт поштучних залишків з WP.xls** 🔜
  WP-вигрузка має колонку «In Stock?» з кількістю по кожному розміру. Зараз
  `parseWp` (`lib/xlsSync.ts`/`catalogImport.ts`) бере лише **назви** розмірів.
  → Розширити парсер: зберігати qty per-size → seed `product_variants.stock_qty`
  + перерахунок дзеркала `products.stock_qty`. Один реімпорт — і весь ERP оживає.
- **A2. Інвентаризація (стоктейк)** ✅
  Документ перерахунку (`ErpStocktake.tsx`, `lib/stocktake.ts`, вкладка ERP
  «Інвентаризація»): додаєш позиції (пошук товару / масово по бренду / усі з
  залишком), вводиш фактичні кількості по розмірах → бачиш розбіжність
  (надлишок/нестача), проведення коригує `product_variants.stock_qty`, логує
  `stock_movements` type='adjust', перераховує дзеркало. Закриває «звірку» з Огляду.
- **A3. Швидке масове редагування залишків** ◻️
  Grid-режим: ввести кількості по багатьох товарах за раз (як Excel), без заходу
  в кожну картку.

## Фаза B — Документи руху (повний складський облік) ◻️

- **B1. Списання (write-off)** — псування/втрата/брак, причини, вплив на залишок
  і фінанси (збиток у P&L). Таблиця `writeoffs` + items, дзеркало як у `receipts`.
- **B2. Повернення постачальнику** — часткове сторно приходу (повертаємо брак),
  зменшує залишок, коригує середню собівартість.
- **B3. Продаж → списання по розміру** — зараз заказ зменшує `stock_qty`-дзеркало;
  довести до списання конкретного **варіанта** (розміру) при фулфілменті заказу,
  з рухом type='sale'. Зв'язок `order_items.variant_id`.
- **B4. Мультисклад / локації** (опц.) — якщо з'являться точки/склади: `locations`,
  залишок per-location, переміщення (transfer).

## Фаза C — Фінанси та P&L (бухгалтерське ядро) ◻️

Рушій собівартості вже є — потрібні звіти поверх нього.

- **C1. Звіт прибутків і збитків (P&L)** — Виручка − COGS − Витрати = Прибуток,
  за період. COGS з `order_items.cost_price` (снапшот) + `expenses`.
- **C2. Маржинальність** — по брендах / категоріях / товарах; найприбутковіші та
  збиткові позиції.
- **C3. Оборотність складу** — inventory turnover, «мертвий запас» (не продається
  N днів) → кандидати на знижку/розпродаж.
- **C4. ABC-аналіз** — товари за внеском у виручку/прибуток (A/B/C-групи).
- **C5. Cash-flow по закупівлях** — скільки грошей у залишку заморожено, план оплат
  постачальникам.

## Фаза D — Закупівлі (purchasing intelligence) ◻️

- **D1. Замовлення постачальнику (PO)** — план закупки → стає приходом при отриманні.
- **D2. Поповнення (replenishment)** — підказки «що докупити»: low-stock × швидкість
  продажів × lead-time постачальника.
- **D3. Прихід з файлу** — імпорт прихідного документа XLS (накладна постачальника),
  не лише ручне введення.
- **D4. Прайс-листи постачальників** — зберігати закупівельні ціни, порівняння,
  історія зміни закупки.

## Фаза E — Канали (розширення E5) ◻️

- **E1. Експорт прайс+залишок** ✅
- **E6. Авто-фід по URL** — `/feed/rozetka.xml`, `/feed/google.xml` що віддаються
  «живими» (майданчик тягне сам за розкладом), замість ручного завантаження.
  Кеш + ревалідація.
- **E7. Імпорт замовлень з майданчиків** — Rozetka/Prom замовлення → у наші `orders`,
  списання залишку, єдина воронка.
- **E8. Синхронізація залишків в обидва боки** — продаж на майданчику зменшує наш
  залишок і навпаки (через API майданчиків, коли будуть ключі).

## Фаза F — Операції та доступ ◻️

- **F1. Штрихкоди / етикетки** — генерація `barcode` (поле вже є у variants), друк
  етикеток, сканер-приймання та сканер-інвентаризація.
- **F2. Ролі та права** — комірник (лише склад), бухгалтер (лише фінанси), власник
  (усе). Розширити `adminAuth`.
- **F3. Журнал дій (audit log)** — хто що змінив (залишки, ціни, собівартість).
- **F4. Друковані документи (PDF)** — накладна приходу, інвентаризаційна відомість,
  акт списання, етикетки. (Стиль як accounting-export.)

## Фаза G — Аналітика та прогнози ◻️

- **G1. Швидкість продажів (sales velocity)** per SKU + прогноз на N днів.
- **G2. Сезонність** — продажі по сезонах (поле `season` вже є).
- **G3. Авто-алерти** — low-stock, повільні продавці, негативна маржа, дрейф залишку
  (дзвіночок в ERP + Telegram через існуючий `lib/notify.ts`).

## Рекомендований порядок

1. **A1 (імпорт кількостей)** — одна зміна парсера оживляє ВСІ цифри ERP. Найвища віддача.
2. **A2 (інвентаризація)** — щоб тримати залишки точними після старту.
3. **B1 + B3 (списання + продаж по розміру)** — повний цикл руху товару.
4. **C1 (P&L)** — головний фінансовий звіт, рушій уже готовий.
5. **D1–D2 (закупівлі + поповнення)** — щоб не залишатись без ходових позицій.
6. **E6 (авто-фіди)** — коли потрібні майданчики без ручної роботи.
7. **F1–F4, G** — операційна зрілість і прогнозування.

> Кожна фаза самодостатня й деплоїться окремо (git push → `bash deploy.sh`).
> Усе на PostgreSQL, схема через `ALTER … IF NOT EXISTS` в `pg.ts` (ідемпотентно).
