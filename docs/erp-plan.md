# Mania Group — Максимальний план ERP

Повна дорожня карта ERP-системи магазину брендового одягу. Прив'язана до реального
стану коду (PostgreSQL, Next.js 16, `/erp`). Кожен модуль позначений статусом:
**✅ зроблено · 🔜 наступне · ◻️ заплановано**.

> Принцип: ERP — єдине джерело правди по **залишках, собівартості та грошах**.
> Вітрина (`/`) і адмінка (`/admin`) — споживачі цих даних.

**⚠️ Головна діра зараз:** поштучні залишки (`stock_qty`) = 0 у всіх XLS-товарів
(вигрузки дають лише «в наявності» + розміри, не кількість). Через це вартість
залишку та обороти = 0. Фаза A це закриває.

## Фаза A — Реальні залишки (фундамент, розблоковує всі цифри) 🔜

Без реальних кількостей дашборд, обороти й точні вигрузки порожні. Найвищий пріоритет.

- **A1. Імпорт поштучних залишків з WP.xls** 🔜
  WP-вигрузка має колонку «In Stock?» з кількістю по кожному розміру. Зараз
  `parseWp` (`lib/xlsSync.ts`/`catalogImport.ts`) бере лише **назви** розмірів.
  → Розширити парсер: зберігати qty per-size → seed `product_variants.stock_qty`
  + перерахунок дзеркала `products.stock_qty`. Один реімпорт — і весь ERP оживає.
- **A2. Інвентаризація (стоктейк)** ✅
  Документ перерахунку (`ErpStocktake.tsx`, `lib/stocktake.ts`, вкладка ERP
  «Інвентаризація»): додаєш позиції (пошук товару / масово по бренду / усі з
  залишком), вводиш фактичні кількості по розмірах → бачиш розбіжність
  (надлишок/нестача), проведення коригує `product_variants.stock_qty`, логує
  `stock_movements` type='adjust', перераховує дзеркало. Закриває «звірку» з Огляду.
- **A3. Швидке масове редагування залишків** ◻️
  Grid-режим: ввести кількості по багатьох товарах за раз (як Excel), без заходу
  в кожну картку.

## Фаза B — Документи руху (повний складський облік) ◻️

- **B1. Списання (write-off)** — швидке списання по варіанту вже є (кнопка «−» у
  картці товару ERP → рух type='writeoff' з нотаткою). ◻️ Залишилось: документ
  масового списання з причинами + вплив на P&L (збиток).
- **B2. Повернення постачальнику** ◻️ — часткове сторно приходу (повертаємо брак),
  зменшує залишок, коригує середню собівартість.
- **B3. Продаж → автосписання по розміру** ✅
  `adjustStock` (lib/orders.ts) тепер size-aware: продаж автоматично зменшує
  `product_variants.stock_qty` потрібного розміру (match по `order_items.variation`),
  логує рух type='sale', перераховує дзеркало; скасування/повернення авто-відновлює
  (type='return'). Fallback на дзеркало для товарів без варіантів. **Нуль ручної
  роботи — склад сам актуальний.** + авто-алерт (див. G3).
- **B4. Мультисклад / локації** (опц.) — якщо з'являться точки/склади: `locations`,
  залишок per-location, переміщення (transfer).

## Фаза C — Фінанси та P&L (бухгалтерське ядро) ✅ (у `/admin` → «Облік»)

Рушій собівартості + звіти поверх нього вже працюють (`AdminFinance.tsx`).

- **C1. Звіт прибутків і збитків (P&L)** ✅ — Виручка − COGS − Витрати, за період.
- **C2. Маржинальність** ✅ — по брендах / категоріях / товарах + ABC.
- **C3. Оборотність складу** ◻️ — inventory turnover, «мертвий запас» (не продається
  N днів) → кандидати на знижку. *(Залишилось: окремий звіт оборотності.)*
- **C4. ABC-аналіз** ✅ — товари за внеском у прибуток (A/B/C-групи).
- **C5. Cash-flow** ✅ — надходження/очікування/повернення/витрати по днях; оцінка
  складу в закупці й рознім. ◻️ Окремо: план оплат постачальникам по відкритих PO.

## Фаза D — Закупівлі (purchasing intelligence) ✅ (ядро готове)

- **D1. Замовлення постачальнику (PO)** ✅ — `ErpPurchasing.tsx`/`lib/purchasing.ts`.
  Документ-план: чернетка → відправлено → отримано/скасовано. **Отримання
  делегує проведення в `receiving.ts`** (createReceipt + addReceiptItem + postReceipt)
  → залишок зростає, рахується weighted-average собівартість, `receipt_id` лінкується
  на PO. Таблиці `purchase_orders` + `purchase_order_items` (ALTER-IF-NOT-EXISTS).
- **D2. Поповнення (replenishment)** ✅ — `ErpReplenishment.tsx`. Варіанти ≤ порогу
  (з налаштувань `low_stock_threshold`) × продажі за 30 днів (`stock_movements` type=sale)
  → рекомендована к-сть `max(поріг×2 − залишок, продано30д)`. Чекбокси + к-сті +
  вибір постачальника → 1 клік `createPurchaseOrderFromLines` (закупка береться з
  `cost_price`). Лінк з Огляду (картка «Поповнення» + «Поповнити →» в low-stock).
- **D3. Прихід з файлу** ✅ — `lib/receiptImport.ts` + `ErpReceiving.tsx` («Прихід
  з файлу»). Завантаження накладної XLS/CSV → `parseImportSmart` (offers/master,
  з AI-fallback) → зіставлення з каталогом (`resolveOfferTargets`), ціна стає
  закупочною → створює **чернетку приходу** (find/create variant) → юзер перевіряє
  й проводить через `postReceipt`. Вибір колонки ціни: базова / акційна(нетто).
- **D4. Прайс-листи постачальників** ◻️ — історія закупівельних цін, порівняння.

> Огляд ERP тепер показує закупівельні лічильники: відкриті PO (чернетки/відправлені)
> + заморожена в них сума (`getPurchasingStats`).

## Фаза E — Канали (розширення E5) ◻️

- **E1. Експорт прайс+залишок** ✅
- **E6. Авто-фід по URL** ✅ — `src/app/feed/[channel]/route.ts`. Публічні живі

- **E8. Синхронізація залишків в обидва боки** — продаж на майданчику зменшує наш
  залишок і навпаки (через API майданчиків, коли будуть ключі).
