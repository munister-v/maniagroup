# Intertop / MG — реальні формати обміну (reverse engineering)

Розбір трьох справжніх файлів від замовника (експорти з кабінету Intertop +
мастер-база MG). Це **ground truth** для моделі даних ERP. Зберігаємо логіку,
але не копіюємо «воду» 1:1.

## ① Шаблон додавання НОВОГО товару — `odezda_*.xlsx`
40 колонок, два рядки заголовків: укр-лейбл + машинний ключ. «Багато зайвого».

| Лейбл | ключ | → наша модель |
|---|---|---|
| Артикул | `article` | offer root (mp-код без розміру) |
| **Заводський артикул** | `factory_article` | `products.factory_article` |
| Код товару | `external_id` | `products.sku` (внутр. код) |
| Штрихкод | `barcode` | `product_variants.barcode` |
| Активність | `active` | `product_variants.active` |
| Кількість | `quantity` | `product_variants.stock_qty` |
| Категорія | `category` | `products.category` |
| Базова ціна | `base_price` | `products.regular_price` / variant.price |
| Акційна ціна | `discount_price` | `products.sale_price` / **variant.sale_price** |
| Зображення | `pictures` | `products.images` |
| Назва (укр/рос) | `product_name[uk/ru]` | `products.name` (беремо укр) |
| Опис (укр/рос) | `product_description[uk/ru]` | `products.description` |
| Колір / Сезон / Бренд / Країна / Колекція / Матеріал … | color/season/brand/country/collection/material | відповідні поля products |
| Тип/Вид/Підвид, Стиль, Принт, Наповнювач, Посадка, Рукав… | group/good_type/podvid/style/print/filler/fit/sleeve | attributes (поки ігноруємо) |

## ② Чистий формат ЦІНИ/ЗАЛИШКИ — `prices.csv` («нічого зайвого»)
Рядок = **торгова пропозиція** (один розмір). Саме цей формат возимо туди-сюди.

```
external_Id, factory_article, barcode, size, offer_code, quantity, base_price, discount_price
EFM221.100.6290, EFM221.100.6290, 16094212680, 45, mp634544, 0, 13650, 8600
AMUW22D205_NAV,  AMUW22D205_NAV,  24944-42,    42, mp882301, 0, 7560,  4763
```

- `offer_code` (mp…) — унікальний ідентифікатор **оффера** (= variant), не товару.
- `quantity` майже скрізь 0, max 1 → модель «розмір присутній ⇒ в наявності».
- `base_price` + `discount_price` живуть **на рівні оффера** → variant потребує `sale_price`.
- 4722 оффери ≈ наші product_variants.

→ Експорт (`/erp` Канали) і майбутній імпорт мають говорити саме цими 8 колонками.

## ③ Мастер-база MG — `Остатки * MG.xls` (cp1251)
Людиночитна база власника. Згруповано по бренду (рядки-підзаголовки бренда).

```
КОД, АРТИКУЛ, БРЕНД, НАИМЕНОВАНИЕ, «Размеры со всех складов (можно с повторами)»,
Цена базовая, Цена продажи, Состав, Коллекция, Тип, Цвет, Страна производитель
24047, AL067N020_999, AFTER LABEL, Пуховик, " ", 31710, 25368, "90% пух…", Зима 20/21, Женская, Черный, Китай
```

- **«Размеры со всех складов»** — реальне джерело поштучних залишків (повтор розміру = +1 шт). Це закриває фазу A1 (зараз qty=0).
- `АРТИКУЛ` тут = `factory_article`. `КОД` = внутрішній id.
- `Тип` = гендер/призначення (Женская/Мужская), `Состав` = composition.

## Що з цього випливає для схеми
- `products.factory_article` — заводський артикул (є у всіх трьох файлах). ✅ додано + колонка в списку.
- `product_variants.sale_price` — акційна ціна оффера (з prices.csv / odezda). ✅ додано, редагується в картці.
- `product_variants.offer_code` — mp-код оффера для звірки при імпорті. ✅ додано, редагується в картці.
- Картка товару = вкладки **Товар / Торгові пропозиції** (редагування полів + статус + матриця офферів). ✅
- Імпорт `prices.csv` (offers) → оновлення variant.price/sale_price/stock по ланцюгу offer_code → barcode → factory_article+size → sku+size. ✅ `lib/stockImport.ts` + `/erp` «Завантажити товари» (превью → застосувати).
- Імпорт MG.xls (master) → заповнює `factory_article` (міст КОД→sku), ціни, і сіє реальні залишки з «Размеры со всех складов» (повтор = +1 шт). ✅ Той самий імпортер, авто-детект формату.
- Перевірено: MG-превью 6733/6733 збіг (6646 цін, 5057 зі стоком); prices-превью до мосту 11/4722, після застосування MG — матчиться по factory_article.
